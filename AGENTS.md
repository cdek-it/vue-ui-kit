# Правила для оберток (wrappers)

Этот файл задает единые правила для компонентов-оберток в `vue-ui-kit`.
Обертка должна быть тонким слоем над исходным компонентом и не ломать его API без явной причины.

## Цели оберток

- Дать единый DX для команды (предсказуемые пропсы, слоты, поведение).
- Инкапсулировать проектные дефолты и стили.
- Сохранить совместимость с upstream-компонентом (`primevue` и др.).
- Минимизировать объем собственной логики в обертке.

## Обязательные правила

### 0) Применение правил AGENTS

- Правила из этого файла обязательны для каждого изменения и имеют приоритет над временными workaround-решениями.
- Перед финализацией правок выполнять self-check по мини-чеклисту из этого файла и устранять расхождения до merge.
- После каждого значимого набора правок запускать как минимум `type-check` и `lint` (не откладывать проверки только на финальный этап).
- Каждая правка должна рассматриваться как потенциально релизная: не оставлять «временные» состояния между итерациями.
- Временные отклонения допустимы только на этапе диагностики; в итоговом состоянии они должны быть устранены или явно задокументированы.

### 1) Именование и размещение

- Имена: `PBlock<ComponentName>` для prime-блоков.
- Путь: `src/primeBlocks/PBlock<ComponentName>/PBlock<ComponentName>.vue`.
- Экспорт должен быть добавлен в `src/primeBlocks/index.ts`.

### 2) API-совместимость

- Базовые пропсы upstream-компонента наследуются через `extends <Component>Props`.
- Дополнительные пропсы обертки именуются явно и не конфликтуют с upstream API.
- Нельзя удалять/переименовывать upstream-пропсы без миграционного плана.

Рекомендуемый шаблон:

```ts
interface IPBlockX extends XProps {
  itemAs?: string | Component;
}
```

### 3) Прозрачный проброс

- Использовать `v-bind="$props"` для базового компонента.
- Для item/action-паттерна объединять данные через helper-функцию (например, `getItemAttrs`).
- Не дублировать вычисления, которые уже делает upstream-компонент.

### 4) Слоты

- Пробрасывать стандартные слоты upstream-компонента.
- Кастомизацию обертки делать через fallback в слотах, без блокировки пользовательских слотов.
- Подключать только реально существующие слоты (проверять через типы `...Slots`).

### 5) Типизация

- Использовать `script setup` + TypeScript.
- Типы слотов получать из пакета компонента (например, `type XItemSlotProps = Parameters<...>[0]`).
- Не применять `any`, если можно выразить тип через upstream-типы.

### 6) Стили

- Обертка не должна вносить тяжелые стили по умолчанию.
- Локальные стили добавлять только при необходимости и документировать причину.
- Классы должны быть стабильными и предсказуемыми для переопределения.
- Не использовать `!important` в стилях оберток; конфликты решать специфичностью селекторов, порядком подключения и токенами.
- Стили, относящиеся к `PBlock*` (например, `.p-block-*`), размещать в самой обертке (`<style scoped>` + `:deep(...)`), а не в `theme3.0/components/css/*.ts`.
- `theme3.0/components/css/*.ts` использовать только для базового upstream-слоя PrimeVue (токены/базовые селекторы `.p-*`) и как временный bridge до полной миграции на библиотечные обертки.
- В одном `theme3.0/components/css/*.ts` файле не смешивать upstream-селекторы `.p-*` и wrapper-селекторы `.p-block-*`.
- Если визуальная логика компонента полностью перенесена в обертку, соответствующий upstream `*.ts` файл удалить.
- Для общих дочерних menu-компонентов (например, `PBlockMenuItem`) базовый layout и геометрия (`display`, `align-items`, `caption`, `icon/submenu` alignment) должны быть в самом компоненте.
- Обертки (`PBlockMegaMenu`, `PBlockMenubar`, `PBlockPanelMenu`) должны задавать только контекстную дельту через токены/переменные и не дублировать базовый layout child-компонента.

### 7) Документация и проверки

- Для новой обертки добавить Storybook-историю с базовым и кастомным сценариями.
- В docs-snippets и пользовательских примерах использовать импорты из текущей библиотеки (`@cdek-it/vue-ui-kit`) для компонентов, у которых есть `PBlock`-обертка.
- Во внутренней реализации stories (template/meta файлы) допустим локальный импорт (`@/primeBlocks` или относительный путь).
- Если для компонента есть `PBlock`-обертка, сами stories должны рендерить именно `PBlock`-компонент (а не прямой `primevue` аналог).
- Если `PBlock`-обертки для компонента нет, допустим прямой импорт из `primevue`.
- Для каждого значимого набора правок обязательны `type-check`, `lint`, smoke-проверка story.
- Если API отличается от upstream, описать отличие в документации компонента.

### 8) Figma как источник правды

- Для визуальных правок компонентов источник правды - Figma-макет (конкретный `fileKey` + `node-id`).
- Выбор целевого узла в Figma делать через поиск по названию компонента в слоях (`MegaMenu`, `Menubar`, `PanelMenu`).
- Если текущее поведение компонента расходится с макетом, приоритет у макета, если нет документированной технической причины сделать иначе.
- Для задач, завязанных на дизайн, использовать MCP-доступ к Figma и не фиксировать внутренние ссылки на макеты в коде/документации репозитория.
- Ссылку на макет запрашивать у автора задачи только в крайнем случае, когда узел нельзя надежно найти через MCP-поиск.
- Любые осознанные отступления от макета фиксировать в PR-описании и документации компонента.

## Антипаттерны

- Обертка с собственной бизнес-логикой вместо UI-адаптации.
- Скрытое изменение поведения upstream-компонента без документации.
- Проброс несуществующих слотов/пропсов (ломает типизацию и вводит в заблуждение).
- Использование в stories прямых импортов UI-компонентов из `primevue` вместо `PBlock*` из библиотеки, когда обертка уже существует.
- Копипаст больших шаблонов вместо переиспользования `PBlockMenuItem`/общих хелперов.
- Реализация визуала "по памяти" или по скриншоту без проверки целевого узла в Figma.
- Хранение `PBlock`-специфичных стилей в upstream `theme3.0/components/css/*.ts` после появления обертки.
- Дублирование базового layout `PBlockMenuItem` в нескольких обертках вместо единого источника стилей в child-компоненте.
- Использование `!important` для фикса визуальных конфликтов вместо корректной структуры стилей.

## Мини-чеклист на каждую правку

- [ ] Папка и имя соответствуют `PBlock<ComponentName>`.
- [ ] Экспорт добавлен в `src/primeBlocks/index.ts`.
- [ ] Пропсы типизированы через upstream `*Props`.
- [ ] Слоты проброшены и проверены по `*Slots`.
- [ ] Нет лишней логики и тяжелых стилей в обертке.
- [ ] В стилях обертки не используется `!important`.
- [ ] `PBlock`-специфичные стили находятся в обертке, а не в upstream `theme3.0/components/css/*.ts`.
- [ ] Если стили компонента полностью перенесены в обертку, legacy upstream `*.ts` файл удален.
- [ ] Для общих child-компонентов (`PBlockMenuItem`) базовый layout находится в самом child, а обертки задают только контекстные переменные/дельту.
- [ ] Базовые стили общих child-компонентов параметризованы через CSS-переменные, а обертки переопределяют только эти переменные/контекстные токены.
- [ ] Для компонентов с `PBlock`-оберткой docs-snippets и пользовательские примеры используют импорты из текущей библиотеки (`@cdek-it/vue-ui-kit`); во внутренних story-файлах допустим локальный импорт (если обертки нет, допустим `primevue`).
- [ ] Для визуальных правок целевой узел найден через MCP-поиск по слоям/названию компонента (или есть явное объяснение, почему это невозможно).
- [ ] Добавлены/обновлены stories.
- [ ] Пройдены `type-check` и `lint`.
